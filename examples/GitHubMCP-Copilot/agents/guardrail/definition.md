# Tool Pattern: Guardrail

## What It Does

Monitors all active GitHub MCP tool activity in real time, flagging or intervening when activity puts your repositories, tokens, or data at risk. Rules are generated from the risk scanner's findings — not written by hand based on assumptions.

## When to Use It

Always. The guardrail should be the first pattern activated and the last one deactivated. It's the safety layer for everything else.

## When Not to Use It

There's no situation where you should skip the guardrail entirely. If alert volume is creating noise, tune the escalation threshold and rule specificity — don't turn it off.

## How It Works

```
Risk scanner runs against mcp.json and PAT scopes
        ↓
Risk profile generated
        ↓
Guardrail rules built from profile
        ↓
Guardrail monitors all active MCP tool calls
        ↓
Flags or intervenes when rules are triggered
```

## Two Operating Modes

### Alert Mode (recommended starting point)
Logs suspicious activity and surfaces alerts for review. Nothing is automatically stopped. Good for initial deployment while you validate your rule set.

### Intervene Mode
Can block or reroute a tool call when a rule is triggered. Use after validating your rules are accurate and thresholds are well-calibrated. Start in alert mode for at least two weeks first.

## Rule Sources

Rules come from two places:
1. **Risk scanner output** — generated from your `mcp.json` and PAT scope scan
2. **Shared baseline rules** — always active regardless of scanner findings

## Shared Baseline Rules (always active)

- Log all MCP tool calls with timestamps and correlation IDs
- Alert on any unhandled exception that fails silently
- Flag repeated identical failed tool calls (sign of a loop or broken dependency)
- Alert if a tool attempts to access a resource outside its defined scope
- Block any token or credential string in tool output payloads
- Alert if a tool pattern that was inactive suddenly becomes active without a recorded activation event

## Configuration

Key fields:
- `mode` — `alert` or `intervene`
- `rules` — generated from risk scanner output, supplemented manually if needed
- `alert_destination` — webhook URL for alerts (Slack, PagerDuty, email, custom)
- `intervention_actions` — what happens when intervening (block, reroute)
- `escalation_threshold` — alert count within a time window before severity escalates

## GitHub-Specific Rules Generated by Risk Scanner

- `block_hardcoded_pat` — triggered if a token is found hardcoded in config files
- `alert_overly_broad_pat_scope` — triggered when PAT scopes exceed tool requirements
- `alert_admin_scope_on_non_admin_tool` — admin permissions on standard tool calls
- `alert_write_scope_on_read_workflow` — write permissions used in read-only patterns
- `alert_unlisted_endpoint_call` — tool call to an endpoint not in allowed list

## Guardrail for the Guardrail

- Log all guardrail decisions independently
- Do not give the guardrail write access to other tool pattern configs
- Review guardrail logs weekly during the first month

## Risk Profile Integration

After running the risk scanner, `risk-profile.json` is automatically read by the guardrail to populate its rules. Re-run the scanner whenever you change your `mcp.json` or update PAT scopes. See [risk-scanner/output-schema.md](../../risk-scanner/output-schema.md).
