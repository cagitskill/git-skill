// What is it

A dev box that lives on GitHub.

Instead of cloning a repo and wrestling with local setup, you click a button. GitHub builds you a Linux machine pre-loaded with the project's tools and opens it in your editor.

Each codespace runs as a Docker container on a virtual machine hosted by GitHub. When you connect — from your browser, VS Code, JetBrains, or the GitHub CLI — you land inside that container.

Your repository is cloned to /workspaces, and a config file in the repo (devcontainer.json) defines the languages, tools, and extensions everyone gets. Same setup, every time, for every contributor.

Whatever you're running locally — Mac, Windows, Chromebook — your codespace runs Linux. Consistent and disposable.

// Configuration as code

Your environment, defined in the repo.

One file — devcontainer.json — describes the whole setup. Anyone who creates a codespace from this repo gets the same environment.

Define, don't assume.

Pick a base image, add features (Node, Docker-in-Docker, language runtimes), pin VS Code extensions, declare which ports to forward, and run a setup command after the container builds.

  • Same environment for every dev, every PR, every CI run
  • Versioned with the code — reproducible across the whole team
  • No more "what version of Python are you on?"
.devcontainer/devcontainer.json
{ "name": "My App", "image": "mcr.microsoft.com/devcontainers/python:3.12", "features": { "ghcr.io/devcontainers/features/node:1": { "version": "20" }, "ghcr.io/devcontainers/features/docker-in-docker:2": {} }, "postCreateCommand": "pip install -r requirements.txt", "customizations": { "vscode": { "extensions": ["github.copilot", "ms-python.python"] } }, "forwardPorts": [3000, 5432]}

// How teams share Codespaces

Each codespace is yours alone — but the work travels.

A codespace is tied to one GitHub account. You can't hand someone the keys to your running session, but here are four ways the work moves between teammates.

i

A single codespace is scoped to a single user. Two people can't log into the same one. What you share is the environment, the running app, and the editing session — not the VM itself.

01 · Same setup

Identical environments via devcontainer.json

The fastest "share" is the one that's automatic. Commit the dev container config to the repo, and every teammate who opens a codespace gets the exact same OS, tools, extensions, and ports.

→ Drop .devcontainer/devcontainer.json in the repo
02 · Share previews

Forward a port, share the URL

Run your app, copy the forwarded port URL, send it to a teammate. They see exactly what you see — handy for design reviews, QA spot-checks, demos, or "does this look right?" pings.

Private Org Public
03 · Pair program

VS Code Live Share, inside the codespace

Install the Live Share extension and start a session. Teammates join with their own cursors, can edit files, run terminals, and debug together — like Google Docs for code.

→ Extension: ms-vsliveshare.vsliveshare
04 · Org-owned

Codespaces the org pays for and governs

On Team and Enterprise plans, the org can own member codespaces — pay for them centrally, set spend limits, control machine types, regions, retention, and which port visibilities are allowed.

→ Org Settings → Codespaces → Policies
// Prebuilds — deep dive

Eight minutes of waiting, or ten seconds of waiting.

A first codespace on a big repo means cloning, building, installing, and running post-create scripts — minutes you'd rather not spend. Prebuilds collapse it.

Cold startbig repo, no prebuild
~8 min
With prebuildimage cached per region
~10 sec

How prebuilds fix it

01 · Push Code lands on the branch
02 · Actions Runs the devcontainer build
03 · Cache Image stored per region
04 · Boot Codespace starts in ~10s 🚀

Worth it when…

  • Boot time drops from ~8 min to ~10 sec
  • Predictable for live demos, training, classrooms
  • Monorepos and review-on-every-PR workflows
  • Multi-region teams — caches per region

Watch out for…

  • Burns Actions compute on every push to tracked branches
  • Cached image storage adds up per region
  • Need to maintain the prebuild config alongside the dev container
// Security & governance

Built for the enterprise — but configured per org.

Code never lives on a laptop. IP stays in the tenant. And every control below is set at the organization level — even on Enterprise plans.

⚙

Configured at: Org Settings → Codespaces → Policies. Enterprise plans add the umbrella features (Enterprise Managed Users, audit log streaming, enterprise-wide IP allow lists), but the codespace policies themselves live per organization.

Encrypted secrets

User, repo, and org scope. Encrypted at rest, injected at startup, never logged.

IP allow lists

Restrict where codespaces can be accessed from — corporate VPN, office IPs, only.

No code on local laptops

Source stays in your tenant. Nothing to wipe when a contractor offboards.

Org policies

Control machine types, regions, retention windows, and approved base images.

Full audit log

Every codespace lifecycle event — create, start, stop, delete — logged for review.

BYOD-friendly

Contractors and tablet-toting devs work in the cloud, not on a hardened laptop.
