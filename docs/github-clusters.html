Clusters that run code, clusters that deploy it, clusters that host it.

GitHub surfaces three distinct cluster concepts. Each solves a different problem — but all three live under the same platform, policy, and security model.

01 · Actions Runner Clusters
// GitHub Actions at scale

A fleet of runners that grows with your pipeline.

GitHub-hosted runners are the default. For larger scale, custom environments, GPU workloads, or cost control, you run your own — either as standalone VMs or as a Kubernetes-based cluster via the Actions Runner Controller (ARC).

ℹ

Kubernetes hosts ARC. The Actions Runner Controller (ARC) is a Kubernetes operator — you deploy it onto a K8s cluster (EKS, AKS, GKE, k3s, or any CNCF cluster), and it manages runner pods for you. Each runner pod is ephemeral: it registers with GitHub, picks up exactly one job, then terminates. The Kubernetes cluster is the substrate; ARC is the layer that turns it into a GitHub Actions runner fleet.

How the pieces connect

A workflow job sits in the GitHub queue. ARC's controller — running inside your Kubernetes cluster — polls the GitHub API, sees the job, and provisions a runner pod. The runner registers, picks up the job, executes it, and terminates. Ephemeral by default.

  • Ephemeral runners: each job gets a fresh pod, nothing bleeds between runs
  • Autoscaling: scale from zero to hundreds, back to zero
  • Custom images: bring your own container with pre-baked tools
  • Runner groups: restrict which repos or orgs can target which runner pool

↓
⚙ ARC Controller Kubernetes operator

GitHub-hosted vs. self-hosted runners

GitHub-hosted

  • Zero infrastructure to manage
  • Free tier (2,000 min/month on Free plan)
  • Larger runners: up to 64-core on Teams/Enterprise
  • GPU runners available (beta)
  • No custom base image — only GitHub's images
  • Network can't reach private VPC resources
  • Costs add up on high-volume pipelines

Self-hosted via ARC

  • Full control over environment and tools
  • Runs inside your VPC — access private registries, DBs
  • Autoscale from zero: no idle compute cost
  • GPU nodes, custom hardware, spot instances
  • Ephemeral runners — clean state every job
  • You own the infrastructure and upkeep
  • Runner must have outbound access to GitHub
02 · Kubernetes on GitHub
// Deploying to Kubernetes via GitHub Actions

GitHub as your GitOps control plane.

GitHub Actions workflows can deploy to any Kubernetes cluster — EKS, AKS, GKE, or self-managed. The result: every deploy is a pull request, every rollback is a git revert, and every promotion is gated by environment protection rules.

🔑

The modern way to authenticate is OIDC (OpenID Connect) — GitHub issues a short-lived token that your cloud provider (AWS, Azure, GCP) trusts. No long-lived credentials in your secrets. The token expires when the job ends.

🔀 Pull Request merged triggers workflow
↓
⚙ GitHub Actions runner builds image, calls kubectl
↓ OIDC token
☁️ Cloud IAM (AWS / Azure / GCP) verifies OIDC, grants access
↓
⚓ Kubernetes cluster EKS / AKS / GKE / self-managed

What you get

  • Full audit trail: every deploy, who triggered it, what SHA, logged in GitHub
  • Environment gates: require reviewer approval before deploying to production
  • Branch policies: only main can deploy to prod — enforced in the workflow
  • Keyless auth: OIDC means no credentials to rotate or leak
  • Rollback = git revert: revert the commit, workflow reruns with the old manifests
  • Works with Helm, Kustomize, raw manifests — no vendor lock-in

03 · Codespaces Machine Clusters
// Codespaces at the infrastructure layer

A global pool of dev VMs, managed by GitHub.

When you create a codespace, it lands in a regional cluster of VMs managed entirely by GitHub. You pick the machine size; orgs set the ceiling. Prebuild images are cached per region so the cluster can serve a ready environment in ~10 seconds.

Machine types

Every codespace runs on one of these machine types. Orgs can restrict which tiers members may use — preventing accidental spend on 32-core machines for small tasks.

  • Free plan: 2-core and 4-core available; 60 hrs/month included
  • Org-owned: org pays, org sets the allowed machine types via policy
  • Enterprise: same policy controls, plus EMU and audit log streaming

// Security — across all three cluster types

Shared security model, cluster-specific controls.

Whether it's a runner pod, a Kubernetes deploy, or a dev environment — the same GitHub security primitives apply. Each cluster type adds its own layer on top.

Encrypted secrets

All three cluster types consume secrets at user, repo, and org scope. Encrypted at rest, injected as env vars at runtime — never in logs, never in code.

OIDC keyless auth

Runner workflows and Kubernetes deploy jobs can authenticate to AWS, Azure, and GCP via short-lived OIDC tokens. No long-lived credentials needed, anywhere.

Ephemeral runners

ARC runners terminate after every job. Nothing persists between runs — no secrets cached in memory, no artifacts from previous jobs accessible. Clean slate, every time.

Runner groups

Assign self-hosted runners to groups. Control which repos and orgs can use which runner group. Prevents untrusted repos from running on privileged infrastructure.

Full audit log

Every workflow run, codespace lifecycle event, and deployment is logged. Enterprise plans stream audit logs to your SIEM — Splunk, Datadog, and others.

Environment gates

Kubernetes deploy jobs reference a GitHub Environment. Each environment can require named approvers, wait timers, and branch policies before any deploy proceeds.

// Enterprise & org-level governance

Every control lives at the organization level.

Enterprise plans add umbrella features — Enterprise Managed Users, enterprise-wide audit log streaming, cross-org IP allow lists — but the cluster policies themselves are configured per organization, even on Enterprise.

⚙

Where to find it: Org Settings → Actions for runner policies, Org Settings → Codespaces for dev environment policies, and Org Settings → Environments for deployment gates. Each org on your Enterprise manages its own.

Runners

Runner group policies

Create runner groups and restrict them to specific repositories or all org repos. Set whether public repos can use self-hosted runners — critical for security, since a public fork PR could otherwise run code on your private infrastructure.

Runners

Allowed actions & reusable workflows

At the org level, restrict which GitHub Actions can be used: only GitHub-owned, only verified creator, or a specific allow list. Prevents supply-chain attacks from untrusted third-party actions in the ecosystem.

Kubernetes

Environment protection rules

Every Kubernetes deploy that references a production environment can require named reviewers, a wait timer, and branch restrictions. Adds a human gate to every production deployment, fully audited.

Kubernetes

Deployment history & rollback

GitHub stores the full deployment history for every environment. See exactly what SHA is live in production, who deployed it, and when. A rollback is a re-run of the previous workflow with an older SHA.

Codespaces

Machine type & region policies

Orgs control which machine types members may create — prevent runaway spend by blocking 16-core and 32-core machines for everyday dev. Also restrict codespaces to specific regions for data residency compliance.

Codespaces

Spend limits & billing controls

Org-owned codespaces charge to the org. Set a monthly spend limit — default is $0, which means members can't use codespaces until an admin explicitly enables and funds them. No surprise bills.

Enterprise

Enterprise Managed Users (EMU)

With EMU, every member account is provisioned and controlled by your identity provider (Okta, Azure AD). Members can't create personal GitHub accounts — all activity is on org-owned accounts, all access is revoked at offboarding automatically.

Enterprise

IP allow lists & audit log streaming

Restrict access to your org — codespaces, repos, and the GitHub UI — to an approved set of IP ranges (VPN, office). Stream audit events to Splunk, Datadog, or any SIEM via webhook or cloud storage in real time.

04 · GHES Cluster Architecture
// GitHub Enterprise Server

Its own architecture — entirely separate from GitHub.com.

GHES is not a feature of GitHub.com — it's a self-contained product you install and operate. When you run it as a cluster, you're managing a set of VM-based nodes yourself, whether those VMs sit in a public cloud or inside your own data center.

🏗

GHES clustering is VM-based at every layer — there are no managed Kubernetes abstractions hiding underneath. Every node (primary, replica, cache) is a VM you provision, whether in AWS/Azure/GCP or on bare metal in a DC. You own the OS, the networking, and the upgrade cycle.

Cluster node roles

A GHES cluster is built from specialized VM nodes. Each role can run on its own VM (or set of VMs) — you scale horizontally by adding nodes of the right type, not by resizing a single machine.

  • Primary node: runs the main application, admin console, and coordinates writes
  • Git replica nodes: serve read traffic for git operations — scale these for large teams
  • Storage nodes: hold repository data, LFS objects, and attachments
  • Cache replica nodes: geo-distributed read cache for globally distributed orgs
  • MySQL replicas: database read scaling and HA failover
  • Elasticsearch nodes: code search and issue search indexing
🖥 Primary node VM application + admin console
↓ replication
📂 Git replica VMs read scale · HA
💾 Storage node VMs repos · LFS · blobs
🔍 Elasticsearch VMs code + issue search
🗄 MySQL replica VMs DB read scale · failover

Where the VMs live — two deployment models

Cloud-hosted VMs

  • VMs in AWS (EC2), Azure (VMs), or GCP (Compute Engine)
  • Use cloud load balancers, managed disks, and VPC networking
  • Snapshot-based backup and DR is straightforward
  • Scale storage nodes by attaching larger EBS / managed disk volumes
  • Can use cloud-native monitoring (CloudWatch, Azure Monitor)
  • Still fully self-managed — GitHub doesn't operate it for you
  • GHES must be able to reach GitHub.com for license validation (unless air-gapped)

Self-hosted in a data center

  • Runs entirely on your own hardware — bare metal or on-prem hypervisor (VMware, Hyper-V, KVM)
  • Full air-gap support — no outbound internet required
  • Meets strict data residency requirements (financial, government, defence)
  • Can integrate with on-prem LDAP, SAML, and internal CAs
  • You handle all hardware procurement, capacity planning, and failure domains
  • Upgrades require manual intervention — no auto-patching

How GHES differs from GitHub.com

Architecture

No shared infrastructure

GitHub.com is a multi-tenant SaaS. GHES is a single-tenant appliance — every VM, every database, every byte of storage belongs to your deployment. Nothing is shared with other customers or with GitHub.com itself.

Networking

Air-gap capable

GHES can run with no internet access at all. License validation, package registries, and even Actions runner images can all be served from internal mirrors. Required for classified, regulated, or highly restricted environments.

Operations

You run the upgrade cycle

GitHub releases GHES versions on a regular cadence. You download the patch, schedule a maintenance window, and apply it. Unlike GitHub.com — which updates transparently — your GHES version is pinned until you upgrade it.

Actions on GHES

Self-hosted runners only

GHES does not have GitHub-hosted runners. All Actions jobs run on self-hosted runners you provision — including ARC on Kubernetes if you want ephemeral runner pods. You bring the compute; GHES coordinates the jobs.
