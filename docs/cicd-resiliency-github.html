How Self-Hosted Runners Actually Work

01

Download the runner app

GitHub provides a lightweight runner application. You download and configure it on any machine you control — server, VM, or container.

02

Register with GitHub

The runner registers to a repo, org, or enterprise using a token from GitHub. It connects outbound over HTTPS — no inbound ports needed.

03

Runner polls for jobs

Once registered, the runner polls GitHub for queued jobs. When a job matches its labels, the runner picks it up and executes it locally.

04

You route jobs with labels

Assign labels to runners (e.g. self-hosted, linux, prod). Workflows use runs-on to target the right runner.

# Route this job to your self-hosted runner jobs: deploy: runs-on: [self-hosted, linux, prod] steps: - uses: actions/checkout@v4 - run: ./deploy.sh
// What GitHub does NOT do automatically
Replace a runner that goes offline — it queues the job and waits
Scale up when your queue backs up — that's on you
Clean state between jobs — unless you use the --ephemeral flag

Where Things Break

// tap each to expand

01

Runner goes offline

Server crashes or agent stops. GitHub queues the job and waits — indefinitely.

→ Job stays in "Queued" state with no timeout
→ No automatic recovery or failover
→ Team has to manually restart the runner agent
→ All jobs targeting that runner are blocked
02

Only one runner registered

One runner means one job at a time. Queue backs up. Everything waits.

→ Jobs execute serially, not in parallel
→ High-traffic moments create a bottleneck
→ Time-sensitive deploys miss their window
→ Fix: register more runners to the same repo or org
03

State bleeds between jobs

Runners persist by default. Old files, env vars, or packages from job A affect job B.

→ Cached deps mask a real broken dependency
→ Leftover env vars override your workflow values
→ Tests pass locally, fail unexpectedly in CI
→ Fix: use --ephemeral flag when registering
04

No way to route by environment

Without labels and runner groups, any runner can pick up any job — including prod deploys.

→ Dev job runs on your prod runner by accident
→ Sensitive secrets exposed to wrong workloads
→ No isolation between environments
→ Fix: use labels + runner groups to control routing

What GitHub Gives You to Fix It

// these are native GitHub Actions features — no third-party tooling required

👥

Register multiple runners

Add more than one runner to a repo or org. GitHub distributes jobs across all available runners automatically. If one is busy or offline, others pick up the work.

Settings → Actions → Runners → New self-hosted runner
→ No config change needed in your workflow YAML
→ GitHub queues and dispatches jobs automatically
→ Idle runners cost nothing — they just wait
🧹

Ephemeral runners (--ephemeral)

Register with the --ephemeral flag. The runner handles one job then exits cleanly. No state carries over between runs.

./config.sh --ephemeral
→ Each job gets a clean environment
→ No leftover files, caches, or env vars
→ You re-provision the runner after each job (or use a script/systemd to loop it)
🏷️

Labels and runner groups

Assign custom labels to runners. Use runs-on to route jobs only to runners with matching labels. Runner groups (org-level) control which repos can use which runners.

runs-on: [self-hosted, prod, linux]
→ Keep prod runners isolated from dev/staging
→ Prevent sensitive jobs from landing on wrong machines
→ Runner groups available on GitHub Team and Enterprise
🚦

Concurrency controls in workflow YAML

Use the concurrency key to limit parallel runs and cancel stale jobs automatically when a newer commit arrives.

concurrency: cancel-in-progress
→ Prevents duplicate deploys from piling up
→ Cancels in-progress run when a newer push comes in
→ Works on GitHub-hosted and self-hosted equally

Tools in Play

// what each one is and where it lives

Native
🤖

Runner application

The GitHub-provided agent you install on your machine. Polls GitHub for jobs over HTTPS and runs them locally. Supports Linux, macOS, Windows.

Native
🏷️

Runner labels

Tags you assign to runners. Workflows use runs-on to target specific runners. Default labels: self-hosted, OS, and architecture.

Team / Enterprise
👥

Runner groups

Org-level feature. Group runners and control which repositories can access them. Isolate prod runners from dev. Manage at the org or enterprise level.

Workflow YAML
🚦

Concurrency key

Defined in your workflow file. Limits parallel runs per branch or ref. Can cancel in-progress runs when a newer job starts. No runner config needed.

Native
📡

Runner API

REST API to list, delete, and check status of registered runners. Use it to build health dashboards or automate runner registration and cleanup.

Flag
🧹

--ephemeral flag

Pass during runner registration. Runner exits cleanly after one job. Combine with a loop script or service manager to re-register for the next job.
