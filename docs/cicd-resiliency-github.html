How Self-Hosted Runners Actually Work

01

Download the runner app

GitHub provides a lightweight runner application. Download and configure it on any machine you control — server, VM, or container. Supports Linux, macOS, Windows.

02

Register with GitHub

The runner registers to a repo, org, or enterprise using a registration token from GitHub. It connects outbound over HTTPS — no inbound ports required.

03

Runner polls for jobs

Once registered, the runner polls GitHub Actions for queued jobs. When a job matches its labels, the runner picks it up and executes it locally on your machine.

04

You route jobs with labels

Assign custom labels to runners (e.g. self-hosted, linux, prod). Workflows use runs-on to target exactly the right runner.

# Route this job to your self-hosted runner pool jobs: deploy: runs-on: [self-hosted, linux, prod] steps: - uses: actions/checkout@v4 - run: ./deploy.sh
// What GitHub does NOT handle automatically on self-hosted
Replace a runner that goes offline — it queues the job and waits indefinitely
Scale up when your queue backs up — autoscaling is your responsibility
Clean state between jobs — unless you use the --ephemeral flag at registration

Where Things Break

// tap each to expand

01

Runner goes offline

Server crashes or agent stops. GitHub queues the job and waits — indefinitely. No timeout, no fallback.

→ Job stays in "Queued" with no automatic timeout
→ No failover to another runner
→ Team must manually restart the runner agent
→ All jobs targeting that runner are completely blocked
02

Only one runner registered

One runner means one job at a time. Queue backs up. Everything waits. Time-sensitive deploys miss their window.

→ Jobs execute serially, never in parallel
→ High-traffic pushes create an instant bottleneck
→ Fix: register multiple runners to the same repo or org
→ GitHub distributes jobs automatically — no YAML change needed
03

State bleeds between jobs

Runners persist by default. Old files, env vars, or packages from job A can silently break job B.

→ Cached deps can mask a real broken dependency
→ Leftover env vars override workflow-defined values
→ Tests pass locally, fail unexpectedly in CI
→ Fix: register with --ephemeral — one clean job per runner
04

No routing isolation

Without labels and runner groups, any runner can pick up any job — including sensitive prod deploys.

→ Dev workload lands on your prod runner accidentally
→ Sensitive secrets exposed to wrong workloads
→ No environment isolation by default
→ Fix: labels + runner groups enforce strict routing

What GitHub Gives You to Fix It

// native GitHub Actions features — no third-party tooling required

👥

Register multiple runners

Add more than one runner to a repo or org. GitHub distributes jobs across all available runners automatically. If one is busy or offline, the others keep working.

Settings → Actions → Runners → New self-hosted runner
→ No YAML change needed — GitHub dispatches automatically
→ Idle runners cost nothing, they just wait
→ Add as many as you need — there's no cap on runner count
🧹

Ephemeral runners (--ephemeral)

Register with the --ephemeral flag. The runner handles exactly one job then exits. No state carries over. Every job gets a clean slate.

./config.sh --ephemeral
→ Each job gets a completely fresh environment
→ No leftover files, caches, or environment variables
→ Re-provision after each job using a loop script or systemd service
🏷️

Labels and runner groups

Assign custom labels to runners. Use runs-on to route jobs to runners with matching labels. Runner groups (org-level) control which repos can access which runners.

runs-on: [self-hosted, prod, linux]
→ Keep prod runners fully isolated from dev and staging
→ Prevent sensitive jobs from landing on the wrong machine
→ Runner groups available on GitHub Team and Enterprise plans
🚦

Concurrency controls

Use the concurrency key in your workflow YAML to limit parallel runs and automatically cancel stale jobs when a newer commit pushes.

concurrency: cancel-in-progress: true
→ Prevents duplicate deploys stacking up
→ Cancels in-progress run when a newer push arrives
→ Works equally on GitHub-hosted and self-hosted runners

Tools in Play

// all native to GitHub Actions — no third-party dependencies

Native
🤖

Runner application

The GitHub-provided agent you install on your machine. Polls for jobs over HTTPS. Supports Linux, macOS, Windows. No inbound ports needed.

Native
🏷️

Runner labels

Tags assigned to runners. Workflows use runs-on to target specific runners. Default labels: self-hosted, OS, and architecture.

Team / Enterprise
👥

Runner groups

Org-level access control. Group runners and define which repositories can use them. Isolate prod runners from dev. Managed at org or enterprise level.

Workflow YAML
🚦

Concurrency key

Defined in your workflow file. Limits parallel runs per branch or ref. Cancels stale in-progress runs when a newer push arrives. No runner config needed.

Native
📡

Runner API

REST API to list, delete, and check status of registered runners. Build health dashboards or automate runner registration and cleanup.

Flag
🧹

--ephemeral flag

Pass during runner registration. Runner exits cleanly after one job. Combine with a loop script or systemd service to re-register for the next job.
