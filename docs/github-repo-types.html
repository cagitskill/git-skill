// The mental model

Think of it like a parent company.

Before diving into repo types, the easiest way to picture the hierarchy is as a corporate structure: a parent company, its divisions, and the projects each one runs.

Microsoft is the Enterprise account. Xbox, LinkedIn, GitHub, and Microsoft Engineering are orgs under it. Each org has its own repos.

The orgs are independent — Xbox is a gaming division, LinkedIn is a social platform, GitHub is dev tools, MS Engineering is the broader internal engineering org. They mostly run their own show.

But Microsoft as a parent still wants to push policies down to all of them: branch protection, who can create public repos, IP allow lists, audit log streaming. That's set at the enterprise level and inherited by every org underneath.

At the repo level, each org runs the mix that makes sense for them. Xbox shares very little with GitHub. But GitHub and MS Engineering co-develop products like Copilot and Actions — so they keep some internal repos that both can read across the org boundary, while still keeping their own private repos for things only their team should touch. And every org has a few public repos for open source.

// The five flavors

Same Git underneath, very different rules on top.

A repo's behavior depends on two things: who owns it (a personal account, an org, an enterprise) and its visibility (public, private, internal). Combine those and you get the five flavors below.

🌐

Public repo

Anyone on the internet

Open to the world. Read by anyone, contributed to by anyone with permission. The default for OSS.

→ Visibility: public · Owner: any
👤

User-owned private

A personal account

Lives under your GitHub handle. Can invite outside collaborators directly. Can be made public anytime.

→ Visibility: private · Owner: user
🔒

Org-owned private

An organization

Owned by the org. Access goes through teams, base permissions, and per-repo collaborators. The standard team repo.

→ Visibility: private · Owner: org
🏢

Internal repoEnterprise only

An org inside an enterprise

Visible to everyone in the enterprise, invisible to everyone outside. Powers "innersource" — share across the company without going public.

→ Visibility: internal · Owner: enterprise org
⚙️

Enterprise org repo

An org governed by an enterprise

An org's repos when the org sits under an enterprise account. Can be public, private, or internal — with policies set at the enterprise level.

→ Owner: org · Governed by: enterprise
🛡️

EMU-owned repoLocked down

An Enterprise Managed Users tenant

Lives inside an EMU enterprise. Only private or internal — no public. Users provisioned by your IdP, no outside collaboration.

→ Identity: SCIM · External access: blocked

// Deep dive · internal repos

The hidden third visibility most teams forget exists.

Internal is the default for new repos in any org owned by an enterprise. It's the "innersource" sweet spot — but the rules are non-obvious.

Every member of the enterprise gets read access to every internal repo — across every org, even ones they don't belong to.
Default behavior

The new-repo default

For any org owned by an enterprise, internal is the default visibility for new repos. You can change it at the enterprise policy level — set the default to private if you want to discourage accidental cross-org sharing.

Read access

Read floor, not ceiling

Internal repos have a minimum visibility of read for enterprise members — even if your org-level base permission is set to none. The base permission floor doesn't apply.

Forking

Single-level forks only

You can fork an internal repo, but you cannot fork the fork. This is intentional — it keeps access management sane when every enterprise member already has a default read.

Outside collaborators

Mostly invisible to outsiders

Outside collaborators on an org's other repos cannot see internal repos. You can add an outside collaborator to a fork of an internal repo, but only if they also have access to the upstream.

// Deep dive · Enterprise Managed Users

EMU is GitHub with a perimeter wall.

Enterprise Managed Users replaces personal GitHub accounts with corporate-issued ones, provisioned by your IdP. The trade is reach for control.

What changes when an enterprise turns on EMU

Users no longer create their own accounts. Your IdP provisions them via SCIM, with usernames in the format handle_shortcode. Authentication is enforced through SAML or OIDC — there's no password or 2FA stored on GitHub itself.

The accounts that result can only see and contribute to repositories inside your enterprise. They can read public repos on github.com, but they cannot star, fork, comment, follow, sponsor, or open issues outside the enterprise. Their content is invisible to anyone outside the enterprise.

Public repos, public gists, and outside collaboration are off the table. Repos can only be private or internal. This is the design — EMU is for organizations whose security or compliance posture cannot tolerate the public surface area of standard GitHub.

EMU users can…

  • Contribute to private and internal repos inside the enterprise
  • Create their own private user repos (if enterprise policy allows)
  • Fork private/internal repos into their user namespace or another enterprise org
  • Use Copilot Business or Copilot Enterprise if licensed
  • Read public repos on github.com (but only read)
  • Use Codespaces, but only the ones owned by the enterprise
  • Be added as repository collaborators (the EMU equivalent of outside collaborators) to enterprise org repos

EMU users cannot…

  • Create public repos or public gists
  • Be invited to repos or orgs outside the enterprise
  • Fork repos from outside the enterprise
  • Star, watch, follow, comment, or open issues on outside repos
  • Sign up for Copilot Free or Copilot Pro
  • Use GitHub-hosted runners on personal user-owned repos
  • Create workflow templates for Actions
  • Install non-privileged GitHub Apps on their user account
  • See the sudo prompt for sensitive actions (auth happens at the IdP)

// New · data residency on GHE.com

Your enterprise on its own subdomain.GA

A separately hosted Enterprise Cloud instance — same product, different URL, with your code stored in your chosen region. Built on Azure, isolated from public github.com.

Standard Enterprise Cloud github.com/your-enterprise Shared platform · data stored in the US by default
Enterprise Cloud with data residency octocorp.ghe.com Dedicated subdomain · code stored in your chosen region · Azure-hosted

What it actually buys you

Region-locked storage for code and core repository data. Useful when GDPR, Schrems II, or sector regulators require code to stay in-region.

Built-in EMU. All GHE.com enterprises use Enterprise Managed Users by design — there's no "non-EMU" option.

Different network footprint. SSH key fingerprints and IP ranges differ from github.com. Integrations have to be updated.

Action namespaces retire on first use. When a workflow first uses an Action from github.com, that namespace is locked in your tenant — you can't create an org or repo with the same name. (You can release a name back if you need it.)

Enterprise repository policies & rulesets.2024–2025

A growing set of controls that let an enterprise govern repository lifecycle and content — not just permissions, but what can be created, named, deleted, transferred, and pushed.

Stop accidental — or malicious — deletion of repos that contain compliance-relevant history.

Prevent code from being transferred out of organizations or out of the enterprise entirely.

A new ruleset type that limits what can land in a repo. Block pushes that touch sensitive paths like .github/workflows/, or files over a size threshold.

// Security & identity

What protects the code, and where it's configured.

A grab-bag of controls that span repo, org, and enterprise. Each one has a scope — that's where you go to turn it on.

Encrypted secrets

User, repo, org, and environment scopes. Encrypted at rest, injected into Actions and Codespaces at runtime, never logged.

Scope · user / repo / org / env

Secret scanning & push protection

Detects committed secrets and blocks pushes that contain them. Public repos covered free; private repos covered through Code Security.

Scope · repo / org

Branch protection & rulesets

Required reviews, required status checks, signed commits, linear history. Rulesets stack and target multiple branches at once.

Scope · repo / org / enterprise

IP allow lists

Restrict where the API and git can be hit from — corporate VPN ranges, office IPs, known cloud egress. Org and enterprise scope.

Scope · org / enterprise

SSO & SAML enforcement

Require SSO for org access; force SSO authorization on personal access tokens. EMU goes further — SSO is the only auth path.

Scope · org / enterprise

Audit log + streaming

Every meaningful event recorded. Enterprise plans can stream the log to Splunk, Azure Event Hub, S3, Datadog — for retention beyond GitHub's window.

Scope · enterprise · streaming

Base permissions

Org-wide default — none, read, write, or admin — applied to every member on every org repo. Internal repos override the floor to read.

Scope · org

Forking policy

Allow, block, or restrict where forks can land. Per-org, with enterprise override. EMU users can never fork outside the enterprise.

Scope · org / enterprise

Code Security (Advanced Security)

The umbrella for code scanning (CodeQL), secret scanning on private repos, dependency review, and Dependabot custom rules. Per-repo activation; org-billed.

Scope · per repo, billed at org
